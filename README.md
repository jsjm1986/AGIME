## ✨ 一眼看懂 AGIME

### 你会得到什么

- **个人效率提升**：文档整理、网页采集、电脑操作、定时任务都可以交给 AI
- **团队协作升级**：把技能、流程、文档、Agent 和服务入口沉淀成团队资产
- **对外服务能力**：通过生态协作和数字分身，把知识和服务开放给客户、伙伴和访客
- **长期运行能力**：支持后台执行、多步骤推进、持续跟踪和可恢复任务
- **跨软件智能协作**：通过 Agentify 把传统软件和业务系统接成可持续运行的 Agent 应用

### 核心能力

- 📄 **文档处理**：提取 PDF、生成报表、归档资料、版本管理、AI 解读
- ⏰ **自动化执行**：每天、每周、长期监控、按计划执行
- 🖥️ **本地操作**：打开软件、浏览网页、处理本地文件
- 🔒 **隐私安全**：个人版本地优先，企业版可私有化部署
- 🧠 **记忆与语境**：记住个人偏好，也支持团队语境统一
- 🔌 **扩展生态**：通过 MCP 和扩展系统接入更多工具能力
- 👥 **Team Server**：频道协作、团队 Agent、外部用户、治理与审计
- 🌐 **生态协作**：把团队能力发布成 Portal，对外提供统一入口
- 🧑‍💼 **数字分身**：把岗位经验变成对内/对外可治理的 AI 服务
- 🧪 **Agentify**：通过 API 让传统软件获得 Agent 能力，并支持跨软件协作

### 为什么它和传统 AI 不一样

| 维度 | 传统 AI 服务 | AGIME |
|------|------|------|
| **工作方式** | 主要回答问题 | 既能回答，也能推进任务和形成交付物 |
| **文件与系统** | 通常隔着云端 | 可结合本地文件、网页、软件和业务系统 |
| **持续执行** | 会话结束就结束 | 可定时执行、后台运行、长期监控 |
| **团队协作** | 多为个人使用 | 可沉淀为团队资产和对外服务入口 |
| **对外服务** | 很难直接上线 | 可做 Portal、数字分身、外部用户入口 |
| **企业治理** | 边界和审计较弱 | 支持权限、语境、治理、审计和持续运营 |

### 适合谁

- 想让 AI 真正参与日常工作的个人用户
- 想沉淀标准流程和知识资产的团队
- 想把服务入口、FAQ、资料和对外互动统一起来的企业
- 想把传统软件和业务系统升级成 Agent 协作应用的组织

## 👥 Team Server：把个人 AI 升级成团队协作平台

AGIME 的服务器版不是简单地把聊天记录搬到云端，而是把团队的知识、流程、Agent、资料、对外服务和治理能力放到一个统一工作台里。  
它更像一个“AI 协作中台”：既服务一线业务同事，也服务管理者、项目负责人和企业客户。

> **一句话理解**
>
> `桌面版 AGIME` 更适合个人使用。  
> `Team Server` 更适合把 AI 能力沉淀成团队资产、对外服务入口和长期运营系统。

### 它主要解决什么问题

- 团队经验只在个人电脑里，别人接不起来
- AI 能力分散在不同人手上，无法标准化复用
- 对外服务、客户答疑、资料发布缺少统一入口
- 想让 AI 长期执行任务，但又担心权限、边界和审计

### 现在的服务器版能做什么

| 模块 | 面向业务能得到什么 |
|------|------|
| **对话频道** | 不只是一对一聊天，还能在团队频道里公开/私密协作、线程回复、指定默认 Agent |
| **Agent 与任务** | 团队 Agent 可以持续对话、后台执行、多步骤推进复杂目标 |
| **文档中心** | 上传、归档、版本管理、AI 工作台、资料接收与协作使用 |
| **工具库** | Skills、Recipes、Extensions 统一沉淀、共享、升级 |
| **智能日志** | 用更容易读懂的方式看团队动态、资源变化和 AI 洞察 |
| **生态协作** | 把团队能力、资料和服务 Agent 封装成 Portal，对外提供统一服务入口 |
| **实验室** | 用 Agentify 把 API 资料变成可长期运行的 Agent 应用 |
| **数字分身** | 批量创建和治理对内/对外分身，适合岗位化、标准化服务 |

| **团队管理** | 成员、分组、团队语境、资料解读策略、安全边界统一配置 |

### Harness V4：新的任务执行内核

服务器版现在已经升级到 **Harness V4** 任务内核。  
对大多数用户来说，你不需要关心它的实现名称，但它直接决定了团队 Agent 和 Mission 执行时的稳定性、节奏感和复杂任务表现。

它带来的变化可以简单理解为：

- **任务不再只是“往前跑”**：系统会先判断任务更适合一次完成、分阶段推进，还是拆成多个子目标
- **执行方式会自动选择**：面对简单任务时保持轻量，面对复杂任务时会引入更合适的协作路径
- **支持有边界的辅助分工**：当一个目标适合让辅助 Agent 帮忙时，系统会用受控方式拉起子任务，而不是无边界乱扩散
- **支持并行推进复杂目标**：对比、调研、汇总、多产出物任务可以走更适合的并行路线
- **运行中会保留进展记忆**：系统会持续记录已完成项、缺口、阻塞点和下一步动作，减少“做到一半忘了前面做过什么”
- **产出导向更强**：执行不只看对话是否流畅，更看文件、结论、证据、交付物有没有真正形成
- **恢复能力更强**：任务中断、失败或需要继续时，更容易回到正确节点，而不是从头再来

对企业和管理者来说，Harness V4 的价值在于：

- 更适合承接长流程、跨步骤、可审查的任务
- 更适合把一个目标拆成可跟踪的执行路径
- 更适合把“AI 在做什么、做到哪了、为什么卡住”解释清楚
- 更适合与数字分身、生态协作、实验室应用一起形成长期可运营的服务能力

### 哪些任务最适合 Harness V4

如果任务只是一次简单问答，你未必能明显感知到 Harness V4。  
但当任务开始变复杂时，它的差别会很明显。

比较适合的场景包括：

- **多步骤交付任务**：例如“先调研，再汇总，再生成报告，再形成对外版本”
- **多产出物任务**：例如同时要产出 `说明文档 + 表格 + 页面 + 交付摘要`
- **跨系统任务**：例如既要查资料，又要调用接口，再写入结果并做回读确认
- **长时间任务**：例如需要多轮推进、期间可能暂停、继续、修正或等待审批
- **需要证据链的任务**：例如管理者希望看到“做过什么、为什么这样做、结果是否成立”
- **需要受控协作的任务**：例如要让辅助 Agent 参与，但又不能让它无限扩散

### 为什么它会比普通 AI 工作流更稳

很多传统 AI 工具的问题，不是“不会回答”，而是：

- 做到一半忘了前面已经完成什么
- 卡住时不会明确告诉你为什么卡住
- 复杂任务里容易越做越散
- 任务中断后很难接着做
- 最后说了很多，却没有真正形成交付物

Harness V4 的改进方向，正是针对这些问题：

- 它更强调“节点推进”而不是一直闲聊
- 它更强调“交付物是否形成”而不是只给你一段文本
- 它更强调“缺口、阻塞、下一步”是否清楚
- 它更强调“需要并行时并行，不需要并行时保持收敛”
- 它更强调“失败后如何恢复”，而不是让任务直接散掉

所以从用户体验上看，它更像一个能持续推进工作的执行系统，而不只是一个能临时回答问题的聊天 AI。

### 生态协作：把团队能力发布成对外服务

生态协作是服务器版非常重要的一条主线。  
它不是单纯“做个页面”，而是把团队的资料、服务能力和对外入口封装成一个 Portal。

> **一句话定位**
>
> 生态协作 = 把团队知识、服务 Agent 和对外入口做成一个可持续运营的 Portal。

你可以把生态协作理解为：

- 对外知识与服务入口
- 项目交付页、FAQ、客户说明页
- 团队资料的受控开放窗口
- 对外协作、客户沟通、伙伴服务的统一门户

当前生态协作支持的重点能力包括：

- Portal 列表与详情管理
- 访客欢迎语、文档绑定范围、Portal 级技能/扩展边界
- 对外服务 Agent 绑定与接入
- 预览、发布与持续运营
- 把生态协作服务和数字分身服务区分治理，避免边界混乱

一句话说，生态协作适合“对外服务、对外交付、对外展示”。

**更适合谁**

- 有客户、伙伴、渠道或外部协作对象的团队
- 需要把内部资料整理成正式对外入口的团队
- 需要把“服务能力 + 资料边界 + 页面入口”统一运营的团队
- 不希望每次都靠人工重复答疑和解释的团队

**它真正带来的价值**

- 对外信息更统一，减少口径漂移
- 访客看到的内容和权限更清楚
- 服务 Agent、资料范围和技能边界可以一起管理
- 从试运行、内测到正式发布有一条更完整的经营路径

如果你已经有一批内容、知识、FAQ、项目说明、服务入口需要长期对外开放，生态协作会比“单独做个网站”更合适。

### 数字分身：把岗位经验变成可治理的服务入口

数字分身更像“岗位化 AI 服务”。  
它适合客服答疑、售后分流、知识问答、制度检索、流程执行、任务跟进等标准化场景。

> **一句话定位**
>
> 数字分身 = 把一个岗位的服务方式、知识边界和处理流程做成可上线、可治理、可审计的 AI 服务入口。

当前数字分身工作台已经覆盖：

- 创建对外分身和对内分身
- 为分身绑定管理 Agent 和服务 Agent
- 配置文档边界、技能、扩展、权限策略
- 用治理台持续优化能力缺口、风险项和运行表现
- 通过审计中心查看治理轨迹、审批变化与执行记录

如果你要快速开放一个标准岗位，数字分身更直接。  
如果你要做更强的对外页面、SDK 接入或在线编排，再升级到生态协作高级方案。

**从企业使用角度看，它最适合解决这类问题**

- 某个岗位的大量重复答疑和标准处理工作
- 需要明确权限边界、升级路径和责任归属的服务流程
- 希望先把“一个标准岗位”跑顺，而不是一开始就做复杂平台
- 想持续治理，而不是把分身上线后放着不管

**它的价值在于**

- 把岗位经验固化成可复制的服务入口
- 让对内和对外服务可以分开治理
- 让“能力扩张、权限放开、流程优化”都变成可审查动作
- 更适合客服、运营、知识服务、流程跟进这类长期存在的场景

简单说，数字分身更像“岗位服务产品化”；  
生态协作更像“对外入口和服务经营平台”。

### 实验室：Agentify｜万物智能

实验室不是展示区，而是新能力的工作区。  
目前首个可用实验应用是 **Agentify｜万物智能**。

> **一句话定位**
>
> Agentify = 通过 API 把传统软件和业务系统赋予 Agent 能力，并让多个软件之间形成跨软件协同 Agent。

它做的事很清晰：

- 导入 API 资料、OpenAPI、Postman、curl 或接口说明
- 让 AI 先理解一个软件或系统的可调用能力
- 把传统软件、内部系统、SaaS 工具封装成可执行的 Agent 应用
- 让多个软件之间形成跨软件协作，而不是各自孤立工作
- 支持一次运行、按计划运行、长期监控
- 保存运行记录、计划和产物，便于持续维护

这意味着，团队不再只是“调用某个接口”，而是可以把多个软件和系统连接成一个可对话、可执行、可长期运行的协作型 Agent App。

**如果从业务视角看，Agentify 适合这些场景**

- 想把多个 SaaS、内部 API、业务系统连成一个连续工作流
- 想把“接口能力”变成业务人员可直接使用的 Agent 应用
- 不想每次都手工拼接接口、脚本和临时流程
- 希望一个应用既能聊天驱动，也能定时执行，还能长期监控

**它的意义不是“多接一个 API”**

- 而是让传统软件通过 API 获得 Agent 能力
- 让多个软件之间能共同完成一个业务目标
- 让原本分散的系统能力变成一个可运营、可持续演进的智能应用

### 对话频道：把多人协作和 AI 协作放在同一个空间

服务器版对话已经不只是个人聊天窗口。

现在你可以：

- 在团队里发起个人对话或进入团队频道
- 创建公开频道和私密频道
- 为频道指定默认 Agent
- 在消息下继续线程讨论
- 让 AI 直接参与频道里的上下文协作
- 在普通对话中保存“个人记忆”，让助手更懂你的称呼、角色、关注点和沟通偏好

这让团队协作从“一个人和 AI 对话”升级成“多人在同一个 AI 上下文里协作”。

### 外部用户：让客户和伙伴有独立入口

服务器版现在还支持外部用户工作台。  
这类用户不需要进入你的内部团队空间，但可以拥有自己的访问账号和使用记录。

对企业来说，这意味着你可以：

- 为客户、合作伙伴、渠道方单独开账号
- 按状态启用或停用
- 重置密码、收回访问
- 查看外部用户的互动和事件记录

这比“把内部系统直接给外部人看”更可控，也更适合正式业务场景。

### 企业管理者最关心的能力

如果你是团队负责人、部门管理者或企业 IT/运营负责人，当前服务器版已经有一批很实用的管理能力：

- **团队语境统一**：为普通聊天窗口设置公司名、部门名、团队简介、业务语境和沟通风格
- **资料解读策略**：统一配置文档自动分析与 AI 资源解读
- **默认团队 Agent**：让团队成员进入后默认使用同一套服务能力
- **权限与边界治理**：将对话、数字分身、生态协作和外部用户分开治理
- **审计与回看**：在数字分身治理、外部用户事件等场景中保留可追踪记录
- **账号与接入管理**：支持内部成员协作，也支持外部用户独立接入

### 典型场景

| 场景 | 怎么用 | 结果 |
|------|------|------|
| **客户服务入口** | 用数字分身创建客服/售后助手，绑定资料和升级到人工边界 | 对外响应更稳定，减少人工重复答疑 |
| **项目对外交付** | 用生态协作发布项目 Portal，统一放说明、FAQ、资料和对话入口 | 客户、伙伴看到同一版本信息，减少反复沟通 |
| **企业内部知识问答** | 用数字分身或团队 Agent 接管制度、流程、知识检索 | 新人更快上手，内部答疑效率更高 |
| **跨系统自动化** | 在实验室里导入 API 资料，用 Agentify 生成长期运行的 Agent 应用 | 多个系统能像一个智能体一样协同工作 |
| **团队频道协作** | 在频道里让成员和 AI 围绕同一主题持续讨论、追踪线程 | 决策上下文集中，减少信息散落 |
| **对外合作管理** | 为外部用户开独立账号，分配边界，跟踪互动记录 | 外部协作更正式、可控、可回溯 |

### 一条更完整的团队使用链路

1. 管理者创建团队、配置默认 Agent、语境和资料策略。  
2. 成员在对话频道、文档中心和工具库里开始协作。  
3. 高价值流程沉淀为 Skills、Recipes 和团队扩展。  
4. 标准岗位通过数字分身快速上线。  
5. 对外服务通过生态协作发布为 Portal。  
6. 跨系统任务通过实验室中的 Agentify 变成可长期运行的 Agent App。  
7. 管理层通过审计、日志和事件记录持续优化协作方式。

## 🏗️ 技术架构

### 本地优先
桌面客户端核心流程本地运行，团队协作通过可选 Team Server 提供集中式能力。

### 服务器版（`agime-team-server`）

- 提供统一团队协作后端与 Web 管理台
- 支持对话、频道、文档、工具库、生态协作、实验室、数字分身、外部用户等模块
- Mission 与团队 Agent 执行已经升级到 Harness V4 内核，适合复杂任务、长流程和持续运行
- 支持多种登录与认证方式
- 正式团队使用建议采用 MongoDB，以获得更完整的服务端能力与持续运行体验
- 适合内网部署、专有云部署和面向企业的统一入口部署

### 通用模型兼容
支持市面上几乎所有主流 AI 模型，包括：
- **云端模型** - 通过 API 接入各大模型服务商
- **本地模型** - 通过 Ollama 等方案完全离线运行

> 💡 你可以根据任务需求和隐私要求，自由选择最适合的模型。

### MCP 扩展协议
基于 Model Context Protocol 标准，支持丰富的插件生态：
- 🔧 **工具扩展** - 文件操作、命令执行、网页浏览
- 🔗 **服务集成** - 连接各类外部服务和 API
- 🎨 **自定义能力** - 开发专属插件满足个性化需求

### 安全沙箱
所有操作在受控环境中执行，支持多级权限管理，确保系统安全。

### 跨平台支持
原生支持 Windows、macOS、Linux，统一的用户体验。
